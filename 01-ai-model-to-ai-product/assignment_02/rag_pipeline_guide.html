§1Why RAG exists

Large language models are frozen on the day they stopped training. RAG is how you teach them about yesterday, about your private documents, and — most importantly — about the things they don't know they don't know.

An LLM trained in 2024 will answer a question about Q3 2025 earnings with the same fluent confidence it answers anything else. That's the problem: the model is always tempted to answer, even when it has no evidence. Worse, its context window has a limit, and even when the limit is huge, answer quality degrades when the relevant fact is buried in the middle of a million tokens (the "lost in the middle" effect).

RAG fixes this by injecting a few carefully selected, highly relevant passages into the prompt at query time, and then instructing the model to answer only from those passages. The model still does the writing — but the facts come from the retrieval step.

The one-liner rule Fine-tune teaches the model to write differently. RAG teaches the model to write about something new. If your problem is "the model needs to know about this 10-K," you want RAG — not fine-tuning.
Course Yuval's analogy: fine-tune = parrot who read many books in a subject. RAG = parrot with a really good search engine.

§3Chunking, in practice

A chunk is the smallest unit you can possibly retrieve. Too small and it's meaningless ("this shall not apply in cases described above"). Too large and a single 8-topic embedding dilutes all 8 topics into mush.

The three levers

chunk_size

How many characters (or tokens) go into one chunk. The assignment fixes this at 1000 chars for Task 3.

chunk_overlap

Characters repeated between neighbours — rescues facts sitting on a chunk boundary. Assignment: 150.

split strategy

Where to break. RecursiveCharacterTextSplitter tries paragraph → line → sentence → word, respecting natural boundaries before falling back to a hard cut.

metadata

Any side info you attach to each chunk (doc_name, page_number, company). Survives splitting and travels with the chunk into retrieval.

Why 150 overlap? Overlap at 10–20% of chunk_size is the textbook range. For chunk_size=1000, 150 is 15% — deep enough to bridge most mid-sentence breaks without wasting index space on duplicates.
Fig 2Chunks inherit page metadata from the Document. The yellow strip is repeated across both chunks so nothing gets severed at a boundary.

Why metadata matters before splitting

If you attach {doc_name, page_number} to the Document before calling the splitter, every child chunk carries it. If you try to do it after splitting you have to recompute it. The assignment insists on page_number as a 0-indexed field so it can be compared directly to the dataset's evidence_page_num (also 0-indexed in FinanceBench).

Gotcha PyPDFLoader already attaches a default page field. You'll either rename it or add a new page_number. Pick one — inconsistency here will haunt Task 6.

§4Embeddings & vector space

An embedding model is a function that maps a string into a point in a high-dimensional space, such that strings with similar meaning land near each other.

For this assignment we use BAAI/bge-small-en-v1.5. Three numbers to remember:

Dimensions384each chunk becomes a vector of 384 floats
Similaritycosinecompares direction only, ignores magnitude
Typical scores[0.6, 1.0]BGE's contrastive training scales scores upward — don't treat 0.5 as "similar"

Why cosine?

If a and b are the query and chunk vectors, cosine similarity is:

$$\mathrm{cos}(a,b)=\frac{a\cdot b}{\lVert a\rVert\,\lVert b\rVert}$$

It's a dot product after length-normalising both vectors. This means "apple pie" and "a long, long essay about apple pie" will land close even though one is much larger in raw magnitude. Dot product would favour the long one. L2 distance mixes geometry with magnitude. Cosine is usually the right default for text.

Fig 3A good embedding clusters semantically similar words. A bad one scatters them — which is exactly what retrieval builds on top of.

Query vs passage symmetry

The bge models are trained so a query embedding and a chunk embedding can be compared directly with cosine. Some embedding families need a query prefix ("query: ...") to work well; bge-small-en-v1.5 recommends it for asymmetric tasks. For FinanceBench — question versus filing prose — the difference is small and LangChain's default behaviour is fine unless you see retrieval doing poorly.

§5FAISS & the index

A vector store is a database tuned for one operation: "given a query vector, give me the k closest stored vectors, fast." FAISS is the one Meta built; it's fast, local, and what LangChain's helper wraps.

At build time you hand FAISS:

  • The chunks (text + metadata)
  • An embedding function

FAISS embeds everything, stores the 384-dim vectors in an index, and keeps the text/metadata alongside. When you later ask for "top-k for this query":

  1. Your query goes through the same embedding model
  2. FAISS runs approximate nearest-neighbour search
  3. It returns k Documents, each with the full metadata you attached at build time
Save it to disk vectorstore.save_local(path) persists the FAISS index + the docstore with metadata. Reloading with FAISS.load_local needs allow_dangerous_deserialization=True. Save it once, reuse it across Tasks 4–7 — re-embedding every notebook restart burns time and API calls.
Numeric scale For FinanceBench's ~40 filings (each ~200–300 pages), chunk_size=1000 gives you roughly 15–25k chunks. Entirely fine for a local FAISS index on a laptop.

§6Retrieval & top-k

Retrieval is the step most RAG pipelines lose on. Get this wrong and it doesn't matter how good your LLM is — it's answering from bad inputs.

The k knob

Top-k is how many chunks you return. Change k and watch two numbers move in opposite directions:

page-hit@k — how the assignment measures retrieval

For each question, FinanceBench tells you the exact page the evidence is on (evidence_page_num, 0-indexed). The metric is binary, per-question:

$$\text{page-hit@}k = \begin{cases}1 & \text{if any of the top-}k\text{ chunks has matching }\texttt{page\_number}\\0 & \text{otherwise}\end{cases}$$

Average across the dataset and you get a single number. Multi-evidence questions (multiple pages) count as a hit if any of those pages is retrieved.

Common bug 0-indexed vs 1-indexed. PyPDFLoader gives 0-indexed. FinanceBench's evidence_page_num is 0-indexed. If you compare 1-indexed to 0-indexed, every hit becomes a near-hit and your number looks mysteriously awful.

Retrieval failure modes, and what to do about each

FailureSymptomFirst thing to try
Right doc not retrievedpage-hit@k low across many questionsRaise k; try a better embedding model; add a reranker
Wrong ranking orderRight page is at rank 5 when k=3Reranker (two-stage retrieve)
Context split across chunksChunk contains half the factIncrease overlap; enlarge chunk_size
Noise drowns signalHigh k but faithfulness fallsSmaller k; reranker; hybrid search

§7Prompting the generator

Retrieval gives you facts; prompting decides how they're used. The exact same chunks fed through two different prompts can give completely different answers.

What a well-formed RAG prompt contains

  1. A system message that sets ground rules: answer only from context, say "I don't know" when it's not there, cite sources.
  2. A context block with each chunk clearly separated and tagged with doc_name + page_number.
  3. The user question, verbatim.
  4. A closing instruction — one sentence reinforcing the grounding rule.
Empty retrieval is a case If the retriever returned nothing, don't hand the model an empty CONTEXT: block. Explicitly tell it "No relevant context was retrieved" and instruct it to refuse. Silently empty blocks confuse models and sometimes trigger hallucinations.

Grounding instructions that actually stick

The assignment's system-prompt requirements distill down to three commandments. Each is there because models fail without it:

1 · Only the context

"Answer the question using only the information in the CONTEXT below." Prevents the model from reaching into its training data.

2 · Admit ignorance

"If the context does not contain the answer, say so — do not guess." Without this, models hedge-hallucinate rather than refuse.

3 · Cite the source

"Each claim must cite the doc_name it came from." Makes faithfulness checkable — and makes the answer itself more useful.

Chunk formatting matters

Think of the context block as a mini-document the model reads. Separators like ---, source headers, and numbered sources all help the model attribute the right claim to the right chunk. A wall of concatenated text without separators makes it harder for the model to reason about where each fact came from.

CONTEXT:
[Source 1 | doc: AAPL_2023_10K.pdf | page: 42]
<chunk text>

[Source 2 | doc: AAPL_2023_10K.pdf | page: 43]
<chunk text>

---
QUESTION: <user question>

Answer using only the above sources. Cite the document for each fact.
If the answer is not present, say so explicitly.

"Garbage retrieval → garbage generation. The best grounding prompt in the world cannot rescue a retriever that missed the page."

§8Evaluating a RAG pipeline

A single end-to-end accuracy number tells you the pipeline is broken but not where. Component-level metrics let you point to retrieval vs. generation and tune the guilty piece.

Fig 4The three evaluation families. Task 6 asks you to run one metric from each column so you can triangulate where failures happen.

The three metrics Task 6 wants

1 · Correctness — LLM as judge

Compare the final RAG answer to the gold answer, using a different model as judge (DeepSeek-V3-0324). The judge returns a binary {correct / incorrect} plus a one-sentence justification. Using a different model than the generator avoids the model "marking its own homework."

Average across the dataset = overall correctness. Simple, but brittle on numeric answers — the judge has to recognise "4.2 billion" and "$4,200,000,000" as the same thing, so the judge prompt matters.

2 · Faithfulness — Ragas

Faithfulness asks a narrower question: is every claim in the answer supported by the context that was actually retrieved? A correct-sounding answer can still be unfaithful if the model pulled the answer from its training data instead of the retrieved chunks.

Ragas works by having a judge model decompose the answer into atomic claims, then checking each claim against the context:

$$\text{Faithfulness} = \frac{\#\text{supported claims}}{\#\text{claims in answer}}$$

Because this is many LLM calls per example, the assignment restricts it to the first 20 questions only.

3 · page-hit@k — retrieval in isolation

Already covered in §6. It's cheap: binary check, no LLM call, run at k ∈ {1, 3, 5}.

Why three metrics, not one High correctness + low faithfulness = model is right by accident (from memory, not from retrieval). High faithfulness + low correctness = model is dutifully quoting the wrong chunks. High page-hit@k + low correctness = retriever did its job, generator dropped the ball. Each combination points to a different fix.
Expect disappointment FinanceBench is hard. Vector-RAG numbers in the literature hover around 19–50% correctness on the full set. Don't chase a benchmark you won't hit — chase insights about which component is failing.

§9Ragas wiring for Nebius

Ragas is a library. Nebius is an OpenAI-compatible API. The assignment wants them glued together with a specific wrench: the collections API + an AsyncOpenAI client + llm_factory. Here's how those pieces relate.

The four pieces you need

PieceWhat it does
AsyncOpenAIOpenAI's async client. Points it at base_url="https://api.studio.nebius.ai/v1/" and it talks to Nebius.
ragas.llms.llm_factoryWraps that client into a shape Ragas metrics accept.
ragas.metrics.collections.FaithfulnessThe new-style Faithfulness metric. Use this, not the older class-based one.
.score(...)Synchronous entry point. Assignment explicitly wants this, not .ascore.

What the call looks like, conceptually

async_client = AsyncOpenAI(api_key=..., base_url="https://api.studio.nebius.ai/v1/")
ragas_llm    = llm_factory(model="deepseek-ai/DeepSeek-V3-0324", client=async_client)
faith        = Faithfulness(llm=ragas_llm)

score = faith.score(
    user_input = question,
    response   = rag_answer,
    retrieved_contexts = [chunk.page_content for chunk in retrieved_chunks],
)
Watch out for Ragas version skew — the .metrics.collections namespace is new. If your import fails, check the installed version (pip show ragas) — you may need to upgrade.

§11Improvement experiments

Task 7 asks for real experimental discipline: one hypothesis per experiment, change one thing from baseline, measure the same three metrics, compare.

The five knobs you can turn

KnobExpected primary effectWatch out for
Generation promptFaithfulness (better grounding instructions)Over-strict prompts → model refuses valid answers
Generation modelCorrectness & faithfulness (smarter reader)Confounds model size with training; don't over-interpret
k in top-kpage-hit@k ↑ with k; faithfulness ↓ if too many distractorsWatch correctness — bigger prompts aren't always better
chunk_sizepage-hit@k (different granularity) & correctnessRequires rebuilding the FAISS index — save under new names
Rerankerpage-hit@k ↑ (correct page moves into top-4)Adds latency; watch the new generation prompt size
One variable at a time If you change both the prompt and k in the same experiment and correctness improves, you won't know which one mattered. Baseline → change 1 → measure → back to baseline → change 2 → measure.

What makes a good hypothesis

A good hypothesis names the metric you expect to move and the reason. Example:

"I expect the reranker to raise page-hit@4 by ~0.1 points because the correct page is often in the top-20 but ranked below rank 4 due to BGE's bi-encoder approximation. Correctness should track upward but more weakly — retrieval isn't the only failure mode."

That's testable. It names the metric (page-hit@4), the expected direction (up ~0.1), and the causal story (bi-encoder mis-ranks within top-20).

Faithfulness subsample The assignment fixes faithfulness on the first 20 questions across all experiments. Do the same 20 every time — otherwise you're comparing apples to differently-ripe apples.

§12Rerankers (cross-encoders)

A bi-encoder (the embedding model) compresses query and chunk into vectors and compares them. A cross-encoder reads both together and scores them directly. The cross-encoder is much more accurate but too slow to run on every chunk — so you use both.

Fig 5Retrieve-then-rerank is the industrial default when bare bi-encoder retrieval is too noisy for the top of the list.

Why keep k the same at the generator?

The assignment is emphatic: when varying k in an experiment, keep the k reaching the generator as your variable; don't also change the retrieve-width. So with a reranker, retrieve top-20 → rerank → keep the top-4 for the generator. The comparison is "4 chunks chosen by bi-encoder alone" vs "4 chunks chosen by bi-encoder-then-cross-encoder."

Cost bge-reranker-base runs on CPU but adds ~100ms per query. For a ~130-question eval run, that's manageable. For production at QPS, you'd GPU it.

§13Multi-scale chunking (bonus)

The hypothesis from the referenced article: no single chunk size is optimal for all queries. A question asking for a single number probably wants tight chunks; a question asking about a 3-paragraph strategic narrative wants wide chunks.

What you're measuring

Build 2–3 FAISS indices with different chunk_size values (e.g., 300, 1000, 2000) and keep everything else identical. For each question, compute page-hit@5 from each index. Then ask:

  1. Per-question winner. For how many questions does the "best" chunk size differ? If it's 10% or less, one winner dominates. If it's 40%+, chunk size is really query-dependent.
  2. Aggregate winner. What is page-hit@5 per index, averaged across all questions? Is one chunk size clearly best on average?
Three possible outcomes (a) One chunk size wins almost always — article's claim is weak here. (b) Different sizes win on different questions, no clear aggregate winner — article's claim is strong; multi-scale retrieval would help. (c) One size wins on average but the disagreement rate is high — article's claim partially holds; worth a hybrid strategy.
